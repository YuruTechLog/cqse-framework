### T-03: 軸は人間が保持するという構造的必然（memo03）

- Cokes三原則草案：
  - 原則1：軸の保持（人間が持つ）
  - 原則2：最小負荷の分業（AIに任せる範囲を最小化）
  - 原則3：責任の帰属（判断の根拠を残す）
- 軸テンプレ（最小版）：目的 / 優先順位 / 抽象度 / 境界条件 / 出力形式

### T-04: 軸語彙5分類と磁力モデル（memo04）
- 軸語彙5分類：起点・抽象度・優先順位・境界・方向性（＋順序=補助）
- 引力モデル：軸は選別ではなく引力（散乱ベクトルを方向に引きつける）
- noteプロンプト集の正体：軸語彙を経験則で寄せ集めた「呪文」

- 磁力強度マップ：方向性 > 境界 > 抽象度 > 起点 > 優先順位 > 順序（条件依存で変動）
- モデル別磁力特性：Claude=抽象度最強 / GPT=起点最強 / Gemini=境界最強 / Copilot=優先順位強

### T-09: Approach Axis命名確定と付属ドキュメント生成（memo09）
- ツール命名検討プロセス：Sync→Align→Contact→Near/Approach→Approach確定
  - Sync：完全同期ニュアンスが強すぎ
  - Align：平行のイメージが違う
  - Contact：「触れたかどうかはわからん」→接触前提が強い
  - Near：状態名で操作性がない（動詞でない）
- Approach確定理由：
  - 動詞→操作できる・失敗を定義できる（Approach Failure）
  - 動的調整ニュアンス：近づく/離れる
  - 一般認知されやすい
  - Cokes「バランス・調和」と整合
- Approach Axisの起源が確定した一言：「人間の軸にも範囲があって、論理和の部分を作ろうぜってかんじから？」
  - 人間の軸=範囲（点ではない） / AI=ベクトル場（場） / Axis=その論理和を扱う道具
- 迎合問題の指摘と解決：摩擦・設計の必要性確認 / Field Axisを別軸から提示
- Approach Failure=ハルシネーション：動詞だから失敗を定義できる（状態遷移モデルへ接続）
- 付属ドキュメント確定（Layer1〜4全文）：
  - axis-words.md（軸語彙体系）
  - magnet-map.md（磁力強度マップ）
  - model-profile.md（モデル別磁力特性）
  - templates.md（テンプレ集）
- 操作プリミティブとLayerの対応関係明確化
- 軽量モデル補足：Haiku/Fable系は全体的に磁力弱い。モデル差=embedding深さ・attention層数・文脈長に依存
