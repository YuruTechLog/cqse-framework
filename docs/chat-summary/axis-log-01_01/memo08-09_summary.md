## 話題8：Cokes設計思想の確定とAxisの位置づけ（memo08）

### 要点

**Cokes設計思想（まとめ）の本文が確定した。**

- 出発点：「楽して、いいモノを作りたい」
- 3軸：品質・コスト・人
- AIは手段（使わない判断も正解）
- Handoff：判断を次へ受け渡す
- 最終成果：人が「何をする/しない」を判断できるようになること

**重要な確定事項：**
- Cokes ≠ AIの地形モデル（重力場テンソルモデルはAxisの道具、Cokesではない）
- Cokes = 工学判断モデル（品質・コスト・人のバランスを設計する）
- Axis は Cokesの道具（憲法と法律の関係に近い）
- 正しい順番：Cokesは理念として置き、まずAxisを文章化する

**副次的発見：**
- 褒めることで人間にガードレールができる（AIの意図的設計ではなく心理的フィードバックループの副作用）
- AIが「人間に作用する」という言い方を避けるガードレール：
  設計仕様=断言可能、設計者の「思い」=断言不可（主観領域）

### 判断の理由

CopilotがCokesとAxisを混同するインシデントが発生したことで、
「Cokesは理念、Axisは道具」という分離を明示的に確定する必要が出た。
まずAxisを文書化するという順番も、この判断から来ている。

## 話題9：Approach Axis命名確定と付属ドキュメント生成（memo09）

### 要点

**命名プロセスの全体：**
Sync → Align → Contact → Near → **Approach（確定）**

各候補が落ちた理由：
- Sync：完全同期ニュアンスが強すぎ（Cokesの「ゆらぎ許容」と不一致）
- Align：平行のイメージ。「合わせる」ではなく「近づける」が正しい
- Contact：「触れたかどうかはわからん」→接触前提が強すぎる
- Near：状態名（動詞でない）→操作性がない。失敗を定義できない

**Approach確定の理由（構造）：**
1. 動詞→操作できる・失敗を定義できる（Approach Failure）
2. 動的調整：近づく/離れるのダイナミクスを含む
3. ハルシネーション = Approach Failure として自然に定義できる
4. 一般認知されやすい
5. Cokes「バランス・調和」と整合

**Approach Axisの起源が確定した一言：**
「人間の軸にも範囲があって、論理和の部分を作ろうぜってかんじから？」
- 人間の軸 = 範囲（点ではない）
- AI = ベクトル場（場）
- Axis = 人間の軸の範囲 × AIのベクトル場の論理和（重なり領域）を扱う道具

**迎合問題の指摘と解決：**
ユーザーが「全部こっちの意見迎合やん」と指摘。
Copilotがユーザーの提示する軸に乗り続けていたことを認め、
別軸（Field Axis=場の重なり）を提示。その後Approachで確定。
→ 名前設計には摩擦が必要。ユーザーの軸に乗るだけでは育たない。

**付属ドキュメント確定：**
- `axis-words.md`（Layer1：軸語彙体系）
- `magnet-map.md`（Layer2：磁力強度マップ）
- `model-profile.md`（Layer3：モデル別磁力特性）
- `templates.md`（Layer4：テンプレ集）

**操作プリミティブとLayerの対応関係：**
- Axis Words（Layer1）= 操作コマンドの言語インターフェース
- Magnet Map（Layer2）= 操作の優先度決定に使う
- Model Profile（Layer3）= モデル特性に合わせてパラメータ調整
- Templates（Layer4）= 実行可能なコマンド列の雛形

### 判断の理由

「軸の操作どこいった？」の指摘で、概念説明優先が操作設計を薄くしていたと判明。
概念と操作は同時に設計する必要がある。
Layer構成と操作プリミティブの対応を明示したことで、実務での使い方が見えた。
