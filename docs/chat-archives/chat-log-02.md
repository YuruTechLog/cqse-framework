## 「チャットしているだけ」に見える問題

もう一つ困るのが、外から見るとサボっているように見えることだ。

AIと設計レビューをしている。
Markdownを毎日大量に作っている。
それを全部読んでいる。
比較している。
判断している。
最後にExcel貼り付け用の資料に変換している。

でも周りから見ると、

「あいつチャットしてるだけじゃん」

に見える。

これはかなり損だ。

従来の仕事では、PowerPointやExcelを開いている時間が長かった。

だから、外から見ても「資料を作っている」と分かりやすかった。

しかしAIを使うと、資料を作る前に思考がかなり進む。

Markdownで整理する。
AIと議論する。
Mermaidで図を作る。
判断ログを残す。
最後にExcelへ変換する。

Excelを触っている時間は短い。

しかし、それは仕事をしていないからではない。

**Excelにする前の思考密度が上がっているだけ**

である。

ここは、組織の評価方法が追いついていない。

---

## Markdownは成果物ではなく、思考のソースコード

自分にとって、Markdownは最終成果物ではない。

思考ログであり、整理メモであり、判断材料であり、設計のソースコードに近い。

昔なら、ホワイトボードに書いて消していたもの。
紙にメモしていたもの。
頭の中で整理していたもの。

それがMarkdownとして残る。

さらにAIと相性が良い。

AIはMarkdownを読める。
編集できる。
構造を変えられる。
比較表にできる。
要約できる。
別観点でレビューできる。

Markdownは、人間にもAIにも読める。

ここが強い。

---

## Excelは納品形式であって、正本ではない

Excelを否定したいわけではない。

納品物として必要なら作る。

ただし、Excelを正本にしたくない。

理由は、Excelは情報モデルを表現するには弱いことがあるからだ。

Excelは表現としては便利だ。
一覧性もある。
顧客に渡しやすい。
管理表にも向いている。

だが、設計の思考そのものは、もっと構造的だ。

要件には階層がある。
業務フローには方向がある。
ERには関係がある。
責務には境界がある。
判断には理由がある。
却下した案にも意味がある。

Excelにすると、それらがフラットになることが多い。

だから、自分の中では、

Markdown / Mermaid / ER / ADR が正本。
Excelはビュー。
Excelは納品形式。
Excelはビルド成果物。

という感覚に近い。

ソフトウェア開発で言えば、

ソースコードがMarkdown。
ビルド成果物がExcel。

だから、納品物でなければExcelは作りたくない。

---

## Mermaidのすごさは、図が描けることではない

Mermaidの価値は、単に図を描けることではない。

本当にすごいのは、

**思考の指向性をテキストとして表現できること**

だと思う。

PowerPointの図形では、線はただの線になりがちだ。

それが依存なのか。
時系列なのか。
責務境界なのか。
データフローなのか。
承認フローなのか。

意味は、人間の頭の中にある。

しかしMermaidでは、

```mermaid
flowchart LR
  要求 --> 業務責務
  業務責務 --> API
  API --> DB
```

のように、方向が明示される。

左から右へ考える。
上から下へ進む。
AがBに依存する。
Bの後にCへ進む。

この指向性をAIと共有できるのが大きい。

AIは画像として図を見るより、Mermaidのテキストを読む方がはるかに扱いやすい。

つまりMermaidは図ではなく、設計構造のDSLに近い。

---

## MermaidはAIと人間が共有できる中間言語

AIとの相性で見ると、Mermaidはかなり強い。

自然言語ほど曖昧ではない。
プログラムほど厳密すぎない。
人間が読める。
AIが編集できる。
Gitで差分が取れる。
レビューできる。

これはかなり大きい。

PowerPointで図を直す場合、AIとの共同編集は難しい。

しかしMermaidなら、

「このノードを追加して」
「例外経路を分けて」
「承認後のフローを追加して」
「この責務を別主体に移して」
「正常系と例外系を分けて」

ができる。

そしてGitで差分が見える。

これは設計作業としてかなり大きな変化だ。

---

## ER図が好きな理由

自分はER図が好きだ。

ただ、正確に言うと、ER図そのものが好きというより、

**ER図的な思考が好き**

なのだと思う。

ER図は判断を強制する。

エンティティを切る。
関係を決める。
多重度を決める。
主従を決める。
履歴を持つか決める。
中間エンティティが必要か考える。
NULLを許すか考える。
ライフサイクルを考える。

ER図を書くと、曖昧なまま逃げにくい。

これは設計者にとって気持ちがいい。

文章では曖昧にできることも、ERにすると途端に曖昧さが露出する。

「これは本当に1対多か？」
「このエンティティは独立して存在できるのか？」
「履歴はどこに持つのか？」
「この関連は業務上の関連か、実装上の関連か？」

そういう問いが自然に出る。

ER図は、単なるデータ構造の図ではない。

**設計者の判断を可視化する図**

だと思う。

## 意図のない固定長フォーマットが嫌い

逆に、意図のない固定長フォーマットは苦手だ。

固定長の表。
意味のない列順。
理由のない並び。
ただのレイアウト。
「昔からこのフォーマットだから」という資料。

そういうものを見ると疲れる。

嫌いなのは、Excelそのものではない。

嫌いなのは、

**意味のない構造**

である。

Markdownの見出しには意味がある。
Mermaidの矢印には意味がある。
ERのリレーションには意味がある。
Gitの差分には意味がある。

一方で、意図なく並んだ表には、意味がない。

意味が人間の頭の中にしかない。

それだとAIとも共有しにくい。

だから、AI時代には「意味が保存される形式」が重要になると思う。

## GitHubを使うと、設計作業が変わる

MarkdownとMermaidをGitHubで管理すると、設計作業の感覚が変わる。

ファイルが残る。
履歴が残る。
差分が見える。
レビューできる。
ブランチを切れる。
PRで議論できる。
戻せる。
比較できる。

これは、設計をコードと同じように扱う感覚に近い。

従来の設計では、

- Excel_v2
- Excel_v2_fix
- Excel_最終
- Excel_最終_本当
- Excel_最終_レビュー反映版

のようになりがちだった。

GitHubを使うと、これがなくなる。

設計変更がdiffで見える。
なぜ変えたかがPRに残る。
レビューコメントが残る。
判断の履歴が残る。

つまり、設計が「成果物」ではなく「育てる資産」になる。

これはかなり大きい。

## AI時代の設計は、ドキュメント作成ではなく知識構築になる

ここまで考えると、AI時代の設計作業は、単なるドキュメント作成ではなくなる。

むしろ、

**知識構築**

に近くなる。

要求を整理する。
業務フローを描く。
責務を分ける。
ERを描く。
選択肢を洗い出す。
メリット・デメリットを比較する。
リスクを整理する。
判断ログを残す。
採用しなかった案も残す。
最後に納品形式へ変換する。

この流れは、かなりソフトウェア開発に似ている。

Markdownがソース。
Mermaidが構造。
ERがデータモデル。
ADRが判断ログ。
GitHubが履歴管理。
AIが編集者・レビュアー・材料係。
ExcelやPowerPointはビルド成果物。

そう考えると、世界がかなり変わる。

## AI時代に重要なのは「情報を作る力」より「情報を咀嚼する力」

AIによって、情報を作るコストは下がった。

文章も出る。
表も出る。
図も出る。
選択肢も出る。
比較も出る。

だから、これから価値が上がるのは、

**情報を作る力**

ではなく、

**情報を咀嚼して判断する力**

だと思う。

AIが出した情報を見て、

- 採用する
- 保留する
- 捨てる
- 分解する
- 別軸で再評価する
- 組み合わせる
- 判断ログに落とす

これができる人が強い。

逆に、AIが出した情報を全部受け取ってしまう人は、かえって混乱する。

AIは非常に優秀な材料係だ。
しかし、材料が増えるほど、調理する人間の能力が問われる。

---

## AIが進化しても、人間の役割は残る

AIがどれだけ進化しても、人間の役割はなくならないと思う。

ただし、役割は変わる。

以前は、人間が情報を集め、資料を作り、比較していた。

これからは、AIがかなりの部分を支援する。

その代わり、人間には、

- 何を問うか
- どの軸で見るか
- 何を捨てるか
- どの前提を採用するか
- どの案に責任を持つか

が求められる。

つまり、人間は「作業者」から「判断者」に寄っていく。

ただし、判断者は楽ではない。

判断密度が上がるからだ。

AIが速くなるほど、人間はより多くの判断を短時間で求められる。

だから、AI時代の働き方は、単純に楽になるわけではない。

むしろ、思考体力が必要になる。

## まとめ：自分にとってAIは、思考の代行者ではない

自分にとってAIは、思考の代行者ではない。

AIに判断を丸投げしたいわけではない。

AIは、

- 整理役
- 材料係
- 壁打ち相手
- 抜け漏れ検出器
- 比較表作成者
- Mermaid編集者
- Markdown整形者
- 設計レビュー補助者

である。

判断するのは人間。

そのために、AIには勝手に軸を変えないでほしい。
前提を隠してほしくない。
文脈を汚染しないでほしい。
今見ている論点に沿って材料を出してほしい。

AIとの共同作業で重要なのは、

「もっと賢いAI」

だけではない。

むしろ、

**人間の思考状態を壊さずに、情報を整理し、選択肢を出し、判断材料を増やしてくれるAI**

が欲しい。

そして、そのAIと相性が良いのは、意味が保存される形式だ。

Markdown。
Mermaid。
ER図。
GitHub。
ADR。
構造化された設計情報。

これらを使うと、設計作業はかなり変わる。

ExcelやPowerPointで清書する前に、設計はもう進んでいる。

資料を作っているのではない。

意味を構造化している。

判断の材料を育てている。

設計のソースコードを書いている。

たぶん、AI時代の設計作業は、ここから大きく変わる。

## 広げられる派生ネタ

### 1. 「AIで楽になる」は半分嘘

AIで調査・整理は速くなるが、人間の判断密度は上がる。
知的労働では、疲労の種類が「作業疲れ」から「判断疲れ」に変わる。

### 2. Excelは悪ではないが、正本にするとつらい

Excelは納品・一覧・管理には強い。
ただし、設計の意味構造を保持する正本としては弱い場面がある。

### 3. Mermaidは図ではなく、思考のDSL

Mermaidの価値は描画ではなく、関係・方向・依存をテキスト化できること。
AIと共有できる「図のソースコード」。

### 4. ER図は判断を強制する

ER図はデータベース設計だけではなく、業務概念の切り分けと責務判断を表す。
曖昧な理解を逃がさない。

### 5. AI時代に強い人は「プロンプトが上手い人」ではない

本質は、情報を咀嚼して、自分の判断軸で取捨選択できる人。
プロンプト技術は手段。

### 6. AIとの共同作業には「コンテキスト汚染対策」が必要

AIが勝手に論点を広げると、人間の思考状態が壊れる。
重要なのは、発散力ではなく、制御可能な発散。

### 7. 設計書は成果物からソースコードになる

Markdown + Mermaid + GitHub によって、設計は「納品文書」ではなく「変更管理される知識資産」になる。

### 8. AI時代の評価は「作業量」では測れない

チャットしているだけに見えて、実際は設計判断を大量に処理している場合がある。
可視化すべきは資料枚数ではなく、判断ログ・論点整理・採用/棄却理由。

### 9. 人間が握るべきもの

AIに任せるもの：整理、洗い出し、比較、整合性確認。
人間が握るもの：評価軸、前提、採用判断、責任。

### 10. 設計作業の未来

Markdownが正本。
Mermaidが構造。
GitHubが履歴。
AIがレビュアー。
Excel/PPTは出力ビュー。
