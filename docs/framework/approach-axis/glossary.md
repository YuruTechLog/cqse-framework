# Approach Axis 用語定義集

framework_draft.md・各章ドラフトで使う用語の統一定義。
用語ブレ防止用。文章化フェーズで迷ったらここを参照。

> **注意:** ここの定義は学術的厳密性より「操作可能な実務定義」を優先する。
> 「重力場」「ベクトル空間」「距離」等は物理学・数学の正確な用語ではなく、
> Approach Axisの動作を説明するための**比喩的操作語彙**として使っている。
> 論文引用時は対応する学術用語（EBM・多様体・確率密度等）を使うこと。

---

## 主体の定義

Approach Axisは「AIを使う人間（設計者）」の視点で書かれている。

| 主体 | 定義 | 役割 |
|------|------|------|
| **設計者（人間）** | AIを使って何かを達成しようとする人 | 軸を保持し、距離操作を設計する |
| **AI** | 意思のない道具。意味空間を持つ言語モデル | 設計者の操作に応じて応答を生成する |

**このフレームワーク内での定義:** AIは「意思のない道具」とする。
確率密度の分布に従って出力を生成する装置として扱う。
（AI意識・理解・意図の哲学的議論はスコープ外）
「AIが考える」「AIが理解する」はこの文脈では使わない（→NG表現テーブル参照）。

**この視点が崩れると全章が崩れる:**
- AIを主体にして読む → 「AIが軸を持つ」誤解に直結
- 「使われる側」の話として読む → 操作設計の話が受け身の最適化論に変わる

Approach Axisは「どう設計するか」のガイドライン。主語は常に設計者（人間）。

## コア用語

### Approach Axis（アプローチアクシス）
人間の判断軸の範囲とAIの応答空間の重なり領域を、距離操作によって動的に作り出す設計ガイドライン。
「近づくという動作を操作化」した点が命名の核。

### 軸（Axis）
**このフレームワーク文脈での定義:** 人間が保持する判断の基準・方向性。幾何学的な軸（座標軸）ではない。
AIのベクトル空間に基準軸は存在しない。「軸」はあくまで人間側が持つもの。

### 軸語彙（Axis Words）
確率密度の偏りを作る語彙。意味空間に対して「距離操作」を行う道具。
プロンプト内に配置することでAI応答を特定方向の近傍に引き寄せる。
（参照: `axis-log-01/axis-words-magnet-map.md`）

### 意味空間（Semantic Space）
AIの内部ベクトル空間。基準軸は存在しない。確率密度の分布として理解する。
「軸がある」ように見えるのは、軸語彙による偏りの結果。

### 距離操作（Distance Manipulation）
軸語彙によって意味空間の確率密度を偏らせ、AI応答を意図方向の近傍に引き寄せる操作。
「狙い撃ち」の実態。偏り誘導とも呼ぶ。

### 偏り誘導（Bias Induction）
距離操作の別名。より正確な表現。
確率密度を「集める」のではなく「傾ける」。点への収束ではなく場の歪曲。

### 人間の軸レンジ（Human Axis Range）
人間が保持する判断範囲。意図・優先順位・許容範囲を含む。
AIの応答空間と重ねる対象。

### 重なり領域（Overlap Zone）
人間の軸レンジとAIの応答空間の論理和部分。
Approach Axisが動的に作り出す対象。ここにAI応答が落ちていれば Success。

## 状態・評価用語

### Approach Failure
ハルシネーションの定義。重力場テンソルモデルでは「地形崩壊」。
軸語彙による傾斜が強すぎた結果、AI応答が人間の軸レンジ外に逸脱した状態。
対処: 即停止 → 根拠トレース → 不可侵領域再強化 → 再設計。
（参照: `axis-log-01/approach-axis-framework.md` §状態モデル）

### 品質スコア
**[未整理 / 定義保留]** 算出方法・閾値根拠は未確定。
現状: 0–100のスケールを想定。閾値 90/70 は仮置き。
Success: ≥90 / Partial: 70–89 / Failure: <70 または ハルシネーション検出。
→ 正式定義は別途設計が必要。このフレームワーク内では「判定の粒度を持つ指標が存在する」という概念として扱う。

### 不可侵領域（Constraint Zone）
AIに絶対に踏み込ませない領域。Failure検出時に再強化する。
プロンプトの `applyConstraint()` で設定。

### Lock
合意領域を固定し、以降変更しない状態。Handoff前に実施。

### Handoff
ゲート付きチェックポイント。成否ゲートを通過した時点で責任が移転する。

- チェックポイント: 状態を評価する場所
- ゲート: 通過条件（Success → 手離れ / Partial・Failure → 差し戻し）
- 手離れ: ゲート通過後、次フェーズへ責任が移る
- 「何をする/しない」が判断できる状態を渡すことが通過条件

**工学的前提（変更の視点）:**
渡した時点の定義・軸レンジ・制約が次フェーズの初期条件になる。
前提が変わった場合 → 新しいHandoffを発行する（前提更新 = 新しいチェックポイントの生成）。
「渡して終わり」は工学的に誤り。

**エージェント設計との接点:** マルチエージェント・LLMワークフロー界隈で「Handoff」は実装レベルで盛んに議論されている。AXISのHandoffは「渡す条件（ゲート）」「通過判定（Success/Partial/Failure）」「責任の所在（人間）」を三点セットで定義する点が特徴。単なる状態受け渡しではなく、運用設計として機能する。

**適用粒度（ゆるてく基準）:**
- プロダクト: 三点セット必須。責任の所在を明確にしないと障害時に詰む
- ファンコンテンツ・個人利用: ゲートを緩めてもOK。「なんとなく投げる」を完全禁止すると運用コストが人間にのしかかる
- 判断基準: 責任を全うできるか（根っこ）→ 把握できるか（手段）→ リカバリーできるか（結果）
  - 丸投げ × 中身把握あり = OK（責任を全うできる）
  - 丸投げ × 中身把握なし = NG（責任放棄）
（参照: `axis-log-03/integrated_knowledge.md` §IK07）

### AIプロファイル
使用モデルの特性定義。軸語彙の効き方がモデルにより異なるため事前に把握する。
例: Gemini=意図方向崩れやすい / Claude=抽象度維持必要 / GPT=起点明確で十分。

## 説明モデル・比喩

### 重力場テンソルモデル（Gravity Tensor Model）
AXIS動作の説明モデル。AI内部=多様体（曲がった地形）/ 軸語彙=傾斜装置 / ハルシネーション=地形崩壊。
論文対応: EBM（エネルギーベースモデル）。
（参照: `axis-log-01/gravity-tensor-model.md`）

### 散弾銃→狙い撃ち
導入比喩。軸なし指示=散弾銃（偶然ヒット）/ AXIS=狙い撃ち（距離操作でヒット率向上）。

### 媒質三分割（Medium Trifurcation）
AI内部の説明モデル。AI=水槽 / 意味空間=水 / パラメータ=水の性質。
重力場テンソルモデルの別角度。
（参照: `axis-log-02/integrated_knowledge.md`）

### 片栗粉モデル（Oobleck Model）
意味空間の操作を2段階で説明する比喩。

- **Stage1（水＋熱 → 粘度変容）**: 軸語彙による緩やかな地形操作。意味空間を傾け、応答を意図方向に誘導する = 正常な距離操作
- **Stage2（衝撃 → 硬化）**: 過剰な制約語彙による地形崩壊。意味空間が固まり応答が動けなくなる = Approach Failure / ハルシネーション

→ 片栗粉は「操作の成功モデル（Stage1）」と「失敗モデル（Stage2）」を両方内包する。

**海モデルとの整合:**
粘度変容（ミクロ・局所操作）→ 海流（マクロ・全体の方向性）に乗る → 意図到達。
衝撃硬化 = 局所に巨大堤防を建てる → 海流が堤防に当たって止まる → 意図した領域に流れが届かない → 前提崩壊。
「守るために建てた堤防が海流を遮断する」= 過剰制約の皮肉をそのまま表す。

**[未整理]** 2段構造の完全統合設計は未完。
（参照: `axis-log-02/integrated_knowledge.md` §片栗粉2段構造）

### 海モデル（Ocean Model）
意味空間のマクロ構造比喩。海流 = 意味空間全体の方向性・流れ。
片栗粉モデル（ミクロ操作）と2層で対応する。

- 水槽（閉じた系）→ 海（開いた系）: スケールアップで系の性質が変わる
- 局所操作（軸語彙）が全体の海流に乗れば意図到達。海流と断絶すると漂流。

**[未整理]** 先行例検索継続中。「世界初」表現は使わないこと。
（参照: `axis-log-02/integrated_knowledge.md` §スケールアップ=性質変化問題）

### 合金比率（Alloy Ratio）
チャンキング粒度の比喩。磁石比喩体系に乗せた表現。

- **比率が粗い（鉄多め）**: 磁力強すぎ → 広域引っかかり → ノイズ混入
- **比率が細かい（銅多め）**: 磁力弱すぎ → 引力不足 → 意図スルー
- **最適比率**: 狙った意味領域だけ引き寄せる

論文対応: 表現学習（Representation Learning）。チャンキング＝バラバラな生データを意味の塊にしてベクトル空間にマッピングする行為。
Attention機構 = 磁力（コサイン類似度）。合金比率 = チャンクの粒度設計。
（参照: `docs/chat-archives/axis-log-04/log12.md`）

### 意味空間歪曲装置（Semantic Space Distortion Device）
Axisの別名定義（log03確定版）。ワープモデルとも呼ぶ。
重力場テンソルモデルと同一概念の別表現。説明対象に応じて使い分ける。
（参照: `axis-log-03/integrated_knowledge.md` §IK03）

### 設計者倫理（Designer Ethics）
責任の所在を明確にする原則。
- 「AIがバカだから」= 責任転嫁
- 正しい認識: 「自分の設計が不足していた」
- AIの限界は仕様。設計者の責任は設計。

**Handoffとの接続:** 責任→把握→リカバリーの構図はここから来る。責任をぼかすと議論全体がぶれる。
**実例:** 新入社員のミスは上司がケツ持ちする。AIに変えても一緒。使う人間が責任者。
　　　　新入社員が独り立ちしたら Handoff。ゲート通過＝責任移転。
（参照: `axis-log-04/integrated_knowledge.md` §KN-MASTER-03）

## 外部参照・関連概念

### CLEAR
形式層のフレームワーク。何をどう書くかを規定。
AXISとの関係: **競合なし。異なるレイヤー。** AXIS設計後にCLEARで形式を整える。
AXIS=どこへ向かわせるか / CLEAR=どう書くか。
（参照: `axis-log-04/integrated_knowledge.md` §KN-MASTER-02）

### Context Engineering
コンテキスト設計の手法体系。システムプロンプト・メモリ・RAG・ツール定義等「AIに渡す情報構造」を設計する。
AXISとの関係: **競合なし。異なるレイヤー。**
Context Engineering=何をどう渡すか（情報構造層）/ AXIS=どの意味空間に向かわせるか（意味介入層）/ CLEAR=どう書くか（形式層）。
3つは積み重なる: AXIS設計→Context Engineering で構造化→CLEAR で形式を整える。
（参照: `docs/chat-archives/axis-log-04/log01.md`・`log03.md`）

### Cokes（コークス）
上位体系。知識整理工学の全体論。Axisを包含する。
関係: Cokes=憲法 / Axis=法律。Axisを Cokesに吸収させると各体系の純度が下がる（NG）。
（参照: `axis-log-01/cokes-design-philosophy.md` §CokesとApproach Axisの関係）

### Fable
人間の未知を扱うTips集。Axisと並列関係。
Axis=AI側の未知を扱う操作体系 / Fable=自分が何を知らないかを確認するTips集。
（参照: `axis-log-03/integrated_knowledge.md` §IK04）

### EBM（エネルギーベースモデル）
重力場テンソルモデルの論文的裏付け。エネルギー地形の最小値方向=AI応答の偏り方向。

### Executor × Advisor協調構造
- Executor（Sonnet等）: 常時稼働・実行・生成担当
- Advisor: オンデマンド呼び出し・骨格抽出・判断基準定義担当
- 人間: 軸を設計・最終判断・責任者

非対称構造（常時稼働 × オンデマンド）。コスト設計: Executor安価・高速 / Advisor高精度・要コスト。
Axisがこの協調を支える構造的基盤。

**注:** モデル名ではなく「役割」として定義する。特定モデルへの依存禁止。
（参照: `axis-log-02/integrated_knowledge.md` §KN18）

## 注意：使ってはいけない表現

| NG表現 | 理由 | 代替 |
|--------|------|------|
| 「AIの軸」 | ベクトル空間に軸は存在しない | 「AI応答空間の偏り」 |
| 「軸を設定する」（AI側に） | 人間側が軸を保持する構造と矛盾 | 「軸語彙で偏りを作る」 |
| 「AIが理解する」 | 理解ではなく確率密度の偏り | 「AIが引き寄せられる」 |
| 「プロンプトを改善する」 | 目的が曖昧 | 「距離操作を設計する」 |
