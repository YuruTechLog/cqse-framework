# knowledge: tide-logs-01 / log03

## テーマ: Cokes6観点による設計検証フレームワーク

### 概要

設計完了後にCokes思想の6観点で照合する品質確認手順。

---

### フレームワーク: Cokes6観点チェック

| 観点 | チェック内容 | 問い |
|------|-------------|------|
| Key Quality | 本来の品質目的を達成しているか | 何を守るための設計か？ |
| Humans | 関わる人間の認知負荷を最小化しているか | 誰が何を選ぶだけでよいか？ |
| Guard | 構造的ガードレールが実装されているか | 誤用を仕組みで防いでいるか？ |
| Handoff | 引き渡しインターフェースが定義されているか | 次の工程に何を渡すか？ |
| Audit | 設定・実行が追跡可能か | 後から何が起きたかわかるか？ |
| 全体 | 理論と実装が一致しているか | 思想が実装に反映されているか？ |

---

### 適用例（今回のケース）

- Key Quality: 免責表示 = 気象情報誤用リスクの事前遮断
- Humans: config.jsonでトーン選択するだけ
- Guard: loader.min.jsが免責を強制挿入
- Handoff: data.json = 解析AI→UIの標準IF
- Audit: config.json + loader.min.jsでトレーサブル
- 全体: Cokes思想と実装が一致

---

### 転用ガイド

- 設計フェーズ後・実装フェーズ前に実施するのが最も効果的
- 一致しない観点があれば設計に戻る（フェーズゲートとして機能）
- 「全体一致」が確認できた時点で設計凍結の判断材料になる
