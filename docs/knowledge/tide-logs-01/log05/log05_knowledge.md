# knowledge: tide-logs-01 / log05

## テーマ: OSSプロダクト向け多層免責防衛設計

### 概要

OSSとしてフロントを公開しながら免責が削除されないための多層防衛パターン。

---

### パターン1: 多層免責構造（5層）

```
Layer 1: UI         → 免責画像 JS自動挿入（本丸・最重要）
Layer 2: data.json  → note + auto_recovery:false（補助・ユーザー不可視）
Layer 3: README     → OSS利用者向け注意書き
Layer 4: LICENSE    → 追加条項（推奨・心理的拘束力）
Layer 5: API        → disclaimerフィールド（fork後も残る）
```

**転用条件**: フロントOSSを公開し、バックエンドを非公開にする構成全般

---

### パターン2: 免責ゲート（未選択=展開停止）

```javascript
const config = await fetch('/config.json').then(r => r.json());
if (!config.disclaimer) {
  showWarningBanner();
  return; // data展開しない
}
loadDisclaimerImage(config.disclaimer);
loadData();
```

**効果**: 免責なし運用を構造的に防ぐ
**転用**: 「設定必須の機能」を持つあらゆるJSコンポーネントに応用可能

---

### パターン3: トーン選択式免責セット

| トーン | 用途 | 特徴 |
|--------|------|------|
| strict | プロダクト向け | 法務的強度最大 |
| neutral | 技術記事向け | 中立・説明的 |
| soft | 個人ブログ向け | 柔らかく圧なし |
| light | SNS向け | カジュアル・最小 |

**config.json**: `"disclaimer": "soft"` で選択
**挿入**: JSが自動で対応トーン画像をDOMに挿入

---

### パターン4: 「やりすぎ停止」判断フレームワーク

**適用条件**: 防衛設計がCokesのHumans（運用者負荷）を圧迫し始めた時点

**判断基準**:
1. すでに最低限の防衛が完成しているか？
2. 追加対応のコストが利益を上回るか？
3. 実証試験のゆるさを損なうか？

3つすべてYesなら停止。

**今回の適用例**: 免責バナー設置領域なし問題 → `document.body` フォールバック1行追加で終了。

---

### 設計原則: 気象業務法対応

- 「予報」ではなく「指標」として提供
- 免責表示を必須にすることで法的保護を確保
- data.jsonに生気象データを含めない（指標値のみ）
- **転用**: 法規制が絡む情報提供サービス全般

---

### 運用パターン: 最小コスト異常検知

- 定期実行の「成功通知」は不要
- 「飛ばない=異常」でSlack通知のみ
- 自動復旧なし宣言 → 文句ゼロ構造
- **効果**: 運用者の心理的負荷を構造的に最小化
