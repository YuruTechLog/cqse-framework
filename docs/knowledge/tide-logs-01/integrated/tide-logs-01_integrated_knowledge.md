## 再利用可能知識: 気象指標サービス設計

### K-01: OSSプロダクト向け免責防衛設計（完全版）

**適用場面**: フロントOSS公開・バックエンド非公開・法的リスクがある情報提供サービス

**構造**:
```
免責画像（トーン別4種）
    ↑ JSが自動挿入
config.json → disclaimer: "soft" 等
    ├── null/未設定 → 警告 + 展開停止（ゲート）
    └── 設定あり → 画像挿入 → data fetch → render

多層バックアップ:
  UI（本丸）/ data.json補助 / README / LICENSE / APIレスポンス
```

**難読化**: 免責挿入ロジックのみ・軽い難読化（変数名短縮・コメント削除）

### K-02: data.json設計原則

**粒度の基準**: UIで実際に使うフィールドのみ（必要十分）

**確定フィールド例（気象指標）**:
- `score`: 数値スコア
- `label`: 評価ラベル
- `summary`: まとめ文
- `reason`: 理由文
- `weather`: 気象値（必要分のみ）
- `updated_at`: 最終更新時刻
- `note`: 補助免責（短文）
- `auto_recovery`: false

**除外**: 生気象データ・中間計算値・デバッグ情報

---

### K-03: 最小コスト運用パターン

```
定期実行（バックエンド）
    ├── 成功 → data.json更新 → Cloudflare配信
    └── 失敗 → Slack通知（「飛ばない=異常」）

人間の作業: Slack確認 → お仕事終わったあとで手動復旧
自動復旧: なし（宣言済み）
```

**効果**: 運用者の心理的負荷ゼロ・文句ゼロ構造

---

### K-04: 「やりすぎ停止」判断基準

以下3条件すべてYesで作業停止:
1. 最低限の防衛はすでに完成しているか
2. 追加コストが利益を上回るか
3. プロダクトのゆるさ・設計者の負荷を圧迫するか

## 再利用可能知識: AIエージェント設計

### K-05: 最小構造イベント駆動エージェント

```
外部イベント（trigger）
    ↓
AI処理（候補生成・分岐分類）
    ↓
チェックポイント（人間介入）
    ↓
人間選択 → 次イベント起動
```

**状態管理**: RAG/DB外付け（Notion型ポーリングは不採用）
**「自走」の実装**: 状態遷移図・目的・イベントポイントを事前設計して外付け

---

### K-06: AI/人間役割分担モデル

| 担当 | 役割 | 強み |
|------|------|------|
| AI | 候補生成・状態解析・大量処理 | 広く見る |
| 人間 | 目的設定・選択・最終判断 | 狭く深く見る |

**境界**: チェックポイント
**適用**: note-pipeline・CI/CD・メール応対・あらゆる自動化システム

## 再利用可能知識: Cokes設計フレームワーク

### K-07: Cokes6観点チェックリスト

| 観点 | 問い |
|------|------|
| Key Quality | 本来の品質目的を達成しているか |
| Humans | 関わる人の認知負荷を最小化しているか |
| Guard | 構造的ガードレールが実装されているか |
| Handoff | 引き渡しIFが定義されているか |
| Audit | 設定・実行が追跡可能か |
| 全体 | 理論と実装が一致しているか |

**使い方**: 設計フェーズ後・実装フェーズ前に実施。不一致があれば設計に戻る。

---

### K-08: 設計アプローチメタパターン

- 「遊びながら構造を組み上げる」→ 制約なし議論 → 後から整理 → 即戦力仕様書
- 個人用途でのオーバースペック排除: 「本当に必要か？」を常に問う
- 「曖昧な言葉（カラー等）は早期確認」: 解釈ズレはインシデントになる
