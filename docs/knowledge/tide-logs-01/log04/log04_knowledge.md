# knowledge: tide-logs-01 / log04

## テーマ: 全自動データ配信システムの実証パターン

### 概要

解析AI→Cloudflare→JSローダー→UIの全自動配信フローの実証済みパターン。

---

### パターン1: 解析AI→UI全自動フロー

```
解析AI（定期実行）
    ↓ data.json生成・更新
Cloudflare Workers KV / R2（配信）
    ↓
loader.min.js（フロント）
    └── fetch → render（設置者無操作）
```

**実証済み**: log04で動作確認
**設置者の作業**: config.json初期設定のみ
**運用中の人手**: 不要（異常時のみSlack通知で検知）

---

### パターン2: data.json必要十分粒度の判断基準

**含めるフィールド**: UIで実際に使うもののみ
**含めないフィールド**: UIで使わない中間計算値・デバッグ情報

**今回の確定フィールド**:
- `score`: 数値スコア
- `label`: 評価ラベル
- `summary`: まとめ文
- `reason`: 理由文
- `weather`: 気象値（潮位・風速・雲量・可視距離）
- `updated_at`: 最終更新時刻

---

### パターン3: バックエンド非公開×フロントOSS分離

**バックエンド**: 解析ロジック・APIキー・生気象データ（非公開）
**フロント**: loader.min.js・config.json・UI（OSS公開）
**境界**: data.json（Cloudflare経由で公開。内容は解析結果のみ）

**効果**:
- ロジック流出リスクゼロ
- フロントをOSSにすることで設置コストをゼロに
- data.jsonの内容で法務リスクも回避（解析結果のみ、生データなし）

---

### 転用ガイド

- 「AI生成JSONをCloudflare経由で配信」パターンはほぼ全分野に転用可能
- config.json設定のみで動く設計は設置者の技術力を問わない
- data.json粒度は「UI要件から逆算する」のが最短
