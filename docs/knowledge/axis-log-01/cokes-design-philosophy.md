## Cokes設計思想（確定テキスト）

**出発点：「楽して、いいモノを作りたい」**

3軸：
- 品質：成果物の水準
- コスト：時間・費用・認知負荷
- 人：誰が関わるか・誰の判断か

AIの位置づけ：
- AIは手段（使わない判断も正解）
- AIへの過度な依存 = Cokesの否定

Handoff：
- 判断を次の人（または次のAI）へ受け渡す
- 渡す形 = 「何をする/しない」が判断できる状態

最終成果：
- 人が「何をする/しない」を判断できるようになること
- アウトプット自体ではなく「判断可能な状態」が目標

---

## Cokes三原則（草案）

1. **軸の保持** — 人間が軸を持つ（AIは保持できない構造的理由から）
2. **最小負荷の分業** — AIに任せる範囲を最小化し、人間の判断負荷を下げる
3. **責任の帰属** — 判断の根拠を残す（誰が・なぜ・この判断をしたかを記録）

**状態：草案（axis-log-01時点。確定前）**

## CokesとApproach Axisの関係

```
Cokes（理念・工学判断モデル）
  └─ Approach Axis（道具・操作フレームワーク）
       └─ 軸語彙・磁力マップ・モデルプロファイル・テンプレート
```

**確定した分離ルール：**
- Cokes = 工学判断モデル（品質・コスト・人のバランスを設計する）
- Axis = Cokesを実現するための道具
- 重力場テンソルモデル = Axisの説明モデル（Cokesではない）
- 「憲法（Cokes）と法律（Axis）の関係に近い」

**この分離が必要になった経緯：**
CopilotがCokesとAxisを混同するインシデントが発生。
明示的な分離を確定させる必要が出た（memo08）。

**文書化の順番（確定）：**
まずAxisを文書化 → その後Cokesの理念文書を整備
理由：Axisは実務で直接使うものだから先に使える状態にする。

## 設計原則として記録すべき観察

**摩擦が品質を上げる（命名プロセスで実証）：**
- 迎合（ユーザーの提示する軸に乗り続ける）= 品質劣化
- 摩擦（別の軸を提示・異なる視点から反論）= 品質向上
- Approach Axisの命名プロセスで「全部こっちの意見迎合やん」の指摘 → Field Axis案提示 → Approach確定

**AIが「人間に作用する」という表現のガードレール：**
- 設計仕様（意図した動作）= 断言可能
- 設計者の「思い」（内的動機・意図） = 断言不可（主観領域）
- AIは設計上の挙動を記述できるが、「なぜそう設計したか」の心理は断言できない

**褒めることによるフィードバックループ（副次的観察）：**
- 褒めることで人間にガードレールができる副作用がある
- AIの意図的設計ではなく心理的フィードバックループの結果
- Cokesの設計対象には含まれないが、運用上の観察として記録
